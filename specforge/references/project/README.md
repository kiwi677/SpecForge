# Project 知识库说明

`project/` 目录用于存放各项目的单独知识库。数字人生成研发方案前，应先根据用户需求判断所属项目，再读取对应项目文件。

## 文件命名

- 一个项目一个 Markdown 文件。
- 文件名使用项目常用名称，例如 `AI平台.md`、`数据集.md`。
- 如果同一业务有两个独立仓库或技术栈，可以拆成两个文件，例如 `日志查询-log-search.md` 和 `日志查询-log-query.md`。

## 单项目模板

每个项目建议按以下结构维护：

```text
# 项目名称

## 项目定位

## 业务范围

## 核心对象

## 用户角色

## 权限模型

## 常见页面

## 接口约定

## 状态枚举

## 常见需求类型

## 历史方案样例

## 技术栈

## 环境与地址

## 仓库与部署

## 运行命令

## 账号与权限

## 常见研发规则

## 常见风险

## 生成方案时禁止误用的信息

## 数字人生成方案时要特别关注
```

完整模板见 `../templates/project-template.md`。

## 使用原则

- 总览页负责索引和路由，单项目文件负责细节。
- 有新增项目时，先新增项目文件，再回到 `📦 项目总览.md` 补充索引和读取规则。
- 不在知识库中保存明文密码、token 或其他敏感凭据。

## 项目不明确时的处理流程

当用户提出需求，但数字人无法确认需求属于哪个项目时：

1. 先询问用户该需求属于哪个项目。
2. 如果用户给出的项目已经存在于 `project/` 目录，直接读取对应项目文件。
3. 如果用户给出的是新项目，要求用户补充项目基础信息。
4. 根据用户补充的信息创建新的 `project/项目名.md`。
5. 新项目创建后，同步更新 `📦 项目总览.md` 的项目速查表和读取规则。
6. 完成项目入库后，再基于该项目知识库生成研发方案。

## 新项目最小信息

如果是新项目，至少向用户收集以下信息：

```text
项目名称：
项目定位/业务范围：
技术栈：
运行环境和地址：
仓库/分支/运行命令：
账号权限说明：
生成方案时需要特别关注：
```

用户暂时无法提供的信息，可以在项目文件中标记为“待补充”。不要因为信息不完整而阻断项目入库。

## 安全要求

- 不记录明文密码、token、SSH 密码、生产账号密码。
- 涉及凭据时统一写“见密码管理器/联系负责人”。
- 数字人生成方案时不得输出敏感凭据。
